Здесь описаны все сценарии Telegram-бота — что бот делает в каждом случае, в каких состояниях находится, какие edge cases обрабатывает.

## Сценарий 0: Создание клуба (CLI, не в боте)

**Где:** на сервере через `docker compose exec bot npm run create-club -- ...`

**Зачем не в боте:** для приватной системы достаточно CLI; UI создания клуба в боте — лишняя сложность. Когда появится веб-фронт, будет сделано там.

**Параметры скрипта:**
- `--name "Четверги у Васи"` — название клуба
- `--host-telegram-id 12345` — Telegram ID будущего host'а
- `--host-nickname "Вася"` — ник в клубе
- `--period week|day|month|manual` — период settle-up (default: manual)
- `--platforms clubgg,pppoker` — используемые платформы (опц.)
- `--host-clubgg-name "Don ron99"` — ClubGG ник host'а (если platforms содержит clubgg)
- `--host-clubgg-id "2700-6386"` — ClubGG ID

**Что делает:**
1. Создаёт `clubs` запись
2. Если user с таким telegram_id ещё нет — создаёт
3. Создаёт `club_members` со status='active', role='host'
4. Если указаны ClubGG данные — создаёт `member_aliases`
5. Шлёт уведомление host'у в Telegram: "Тебе создан клуб «X». Готов добавлять игроков?"

## Сценарий 1: `/start` (онбординг)

**Триггер:** пользователь пишет `/start` (опционально с параметром `/start invite_TOKEN`).

**Логика:**

```
Бот ищет user по telegram_chat_id:

├─ user уже есть в БД:
│  ├─ есть параметр invite → СЦЕНА accept_invite
│  └─ нет параметра:
│     ├─ есть активные membership'ы → главное меню
│     └─ нет membership'ов → "Жди приглашения, либо введи код"
│
└─ user'а нет в БД:
   ├─ есть параметр invite → создаём user → СЦЕНА accept_invite
   └─ нет параметра → создаём user → "Привет! Чтобы пользоваться, нужен инвайт от host'а"
```

**Состояний FSM нет** — это просто маршрутизация на следующий сценарий.

## Сценарий 2: Принятие инвайта (`accept_invite`)

**Триггер:** `/start invite_TOKEN` ИЛИ пользователь жмёт "Ввести код приглашения" в главном меню.

### Шаги

```
Шаг 1. Валидация инвайта (без UI — просто запрос в БД)
   SELECT FROM club_invites WHERE token=? AND used_at IS NULL AND expires_at > NOW()

   ├─ не найден / истёк / использован → "Приглашение недействительно" → выход
   └─ найден → продолжаем

Шаг 2. Проверка существующего членства
   SELECT FROM club_members WHERE user_id=? AND club_id=?
   ├─ уже состоит (status='active') → "Ты уже в клубе «X»" → главное меню
   ├─ есть pending → "У тебя уже подана заявка в этот клуб, ждёшь подтверждения"
   ├─ был rejected → "Тебе отказали ранее. Свяжись с host'ом для нового инвайта"
   └─ нет → продолжаем

Шаг 3. Информация о клубе и приветствие
   "Тебя приглашают в клуб «Четверги у Васи».
    • Host: Don ron99
    • Игроков в клубе: 5
    • Период settle-up: еженедельно

    Войти в клуб?"
   [✅ Да, войти]  [❌ Отказаться]

Шаг 4. (если "войти") Запрос ника в клубе
   "Под каким ником тебя записать в клубе?"
   ─ валидация: 2-64 символа, уникален в (club_id, nickname)
   ─ если занят → "Уже занят, выбери другой"

Шаг 5. (опц., если клуб использует ClubGG) ClubGG-данные
   "Клуб использует ClubGG. Введи свой ник там."
   ─ ввод текста
   ↓
   "Теперь ID в формате XXXX-XXXX"
   ─ валидация regex /^\d{4}-\d{4}$/
   ─ проверка уникальности (source='clubgg', alias_name, alias_id)

Шаг 6. Финальное подтверждение
   "Проверь:
   • Клуб: Четверги у Васи
   • Твой ник: Вася
   • ClubGG: Don ron99 (2700-6386)
   Всё верно?"
   [✅ Да, отправить заявку]  [✏️ Изменить]  [❌ Отмена]

Шаг 7. (если подтверждено) Транзакция в БД:
   - INSERT club_members (status='pending')
   - INSERT member_aliases (если ClubGG)
   - UPDATE club_invites SET used_by_user_id=?, used_at=NOW()
   ↓
Уведомление игроку: "Заявка отправлена. Жди подтверждения host'а."
Уведомление host'у: карточка с [✅ Принять] [❌ Отклонить]
```

### Состояния FSM

```
state: VALIDATING_INVITE
state: AWAITING_INVITE_DECISION
state: AWAITING_NICKNAME
state: AWAITING_CLUBGG_NAME
state: AWAITING_CLUBGG_ID
state: AWAITING_CONFIRMATION
state: COMPLETED → выход в главное меню
```

### Edge cases

1. **Истёкший токен** — "Приглашение истекло" → выход.
2. **Использованный токен** — "Это приглашение уже использовано" → если `used_by_user_id == текущий user`, говорим "Ты уже в клубе".
3. **Один user в один клуб дважды** — UNIQUE `(club_id, user_id)` блокирует.
4. **Ник занят** — повтор шага 4.
5. **ClubGG-ник занят** — "Этот ClubGG-аккаунт уже привязан в системе".
6. **Прерванный онбординг** — state в Redis с TTL 30 мин. При повторном `/start` — "Продолжить или начать заново?".
7. **Отказ на шаге 3** — токен НЕ помечается использованным, можно зайти снова.

### Решения

- ✅ Инвайт живёт **24 часа**
- ✅ Одноразовый (одна ссылка = один человек)
- ✅ После прохождения сцены — **обязательное подтверждение host'ом**

## Сценарий 3: Подтверждение/отказ заявки host'ом

**Триггер:** host получает уведомление о новой заявке (или заходит в `/members` → "Ждут подтверждения").

```
Бот шлёт host'у:
"📥 Новая заявка в клуб
От: Маша (@masha_t)
ClubGG: masha_pro (1234-5678)
Заявка: 2 минуты назад"
[✅ Принять]  [❌ Отклонить]

═══ Ветка "Принять" ═══
   UPDATE club_members SET status='active', status_changed_at, status_changed_by=host
   ↓
   Уведомление игроку: "Host одобрил твоё вступление! Добро пожаловать."
   Уведомление host'у: "Маша добавлена в клуб."

═══ Ветка "Отклонить" ═══
   "Укажи причину отказа (опционально):"
   [Без причины]  / введи текст
   ↓
   UPDATE club_members SET status='rejected', rejected_reason=<текст>
   Уведомление игроку: "Host отклонил твоё вступление. Причина: <текст>"
```

### Edge cases

1. **Host долго не отвечает** — через 48ч worker шлёт напоминание; через 7 дней — авто-отклонение.
2. **Игрок передумал ждать** — пока команды отзыва нет (можно добавить `/cancel_pending_application`).
3. **Кто-то уже принял заявку** (если co-host'ы будут на уровне клуба) — второму нажатию говорим "Уже обработана".

## Сценарий 4: Создание онлайн-сессии (OCR)

**Триггер:** host жмёт `/newgame` → выбирает "🌐 Онлайн".

### Шаги

```
Шаг 0. Проверка: нет ли уже активной сессии у host'а в этом клубе
   SELECT sessions WHERE club_id=? AND host_id=? AND status='in_progress'
   ├─ есть → "У тебя открыта оффлайн-сессия #X. Сначала закрой её." → выход
   └─ нет → продолжаем

Шаг 1. Выбор типа: [🌐 Онлайн] [🪑 Оффлайн]
       (выбран онлайн)

Шаг 2. Название (опц.)
   "Назови сессию (можно пропустить)"
   [Пропустить]  /  ввод текста
   ─ дефолт: "Онлайн-сессия от <дата>"

Шаг 3. Загрузка скринов (главный шаг)
   "Кидай скрины из ClubGG. Когда все загрузишь — нажми 'Готово'.
   ⚠️ Все скрины должны быть от ОДНОГО стола за ЭТУ сессию.
   Если играли за двумя столами — это две сессии."
   [✓ Готово]  [❌ Отмена]

   На каждое присланное фото:
   • Бот моментально: "📸 Скрин #N загружен. Распознаю..."
   • Файл сохраняется в MinIO
   • Создаётся ocr_imports запись со status='pending'
   • Job в BullMQ → worker берёт OCR
   • Через 3-7 сек worker возвращает результат:
       ✅ Скрин #N распознан
       • Стол: NLH bomba (1/2, бай-ин 100~600)
       • Период: 28.04 22:14 — 29.04 10:14
       • Игроков: 7
       • Сумма P&L: 0.00 ✓ (или ⚠️ при расхождении)

       Игроки распознаны:
       ✅ Yakirsneh    → Вася         (+138.59)
       ✅ Don ron99    → ты           (+1176.92)
       ❓ Idanbu3224   → не сопоставлен (-4.41)

   Если OCR упал → шаг "Обработка ошибки OCR"

Шаг 4. (после "Готово") Сопоставление неузнанных игроков
   Для каждого игрока без авто-match:
   "Игрок 'Idanbu3224' (ID 7160-9824)
   Суммарный P&L по скринам: +95.17

   Кто это?"
   [👥 Член клуба ▼]  [👤 Гость]  [❌ Это не игрок]  [⏪ Назад]

   • "Член клуба" → выпадающий список → выбор → создаётся member_aliases для будущего
   • "Гость" → "Имя гостя?" → создаётся session_participants с guest_name (без club_member_id)
   • "Это не игрок" → строка ocr_screen_results не создаётся, P&L игнорируется
                     (на финальном подтверждении баланс может перестать сходиться)

Шаг 5. Финальное подтверждение
   "📊 Итог сессии «Онлайн-сессия от 30.04»
   Скринов: 2  •  Игроков: 7
   Период: 28.04 22:14 — 29.04 10:14

   Итоговый P&L:
   • Вася           +1.47
   • ты             +1801.90
   • Idanbu3224*    +95.17     (* гость)
   • ...
   ─────────────────────
   Сумма:           +0.00 ✓

   Подтвердить и закрыть сессию?"
   [✅ Подтвердить]  [✏️ Исправить]  [❌ Отменить сессию]

Шаг 6. (если подтверждено) Транзакция в БД:
   1. INSERT sessions (status='ended', type='online', started_at=мин period_start, ended_at=NOW())
   2. INSERT tables (распознанные параметры стола)
   3. UPDATE ocr_imports SET status='confirmed' для всех скринов
   4. INSERT session_participants
   5. INSERT ocr_screen_results
   6. INSERT session_results (агрегаты P&L по скринам)
   7. UPDATE club_members.current_balance += session_result.profit_loss
   8. INSERT ledger_entries (по одной на каждого члена клуба, reason='session_result')

Шаг 7. Постановка отложенных уведомлений
   Для каждого участника-члена клуба создаётся notifications запись
   со scheduled_for = NOW() + 5 минут
   ↓
Бот host'у:
   "✅ Сессия записана. Балансы обновлены.
   Игроки получат уведомления через 5 минут.
   Если заметишь ошибку — успеешь отменить."
   [↩️ Отменить сессию (5 мин)]  [📊 Главное меню]
```

### Обработка ошибки OCR

Если worker вернул `status='failed'` для скрина:

```
"❌ Скрин #N не распознан.
Возможная причина: <error_message>

Что делать?"
[🔄 Переснять и загрузить заново]
[✏️ Ввести данные вручную]
[🗑 Удалить этот скрин из сессии]
[❌ Отменить всю сессию]
```

**Ручной ввод** — мини-сцена: бот по одному игроку спрашивает "имя", "ID (опц.)", "P&L". Создаёт `ocr_screen_results` без ссылки на скрин (`ocr_import_id` будет проставлен на саму запись `ocr_imports` со status='confirmed' и пометкой `parsed_data: {manual: true}`).

### Состояния FSM

```
state: AWAITING_GAME_TYPE
state: AWAITING_SESSION_TITLE
state: AWAITING_SCREENS
state: PROCESSING_SCREEN          ← скрин в очереди OCR
state: AWAITING_PLAYER_MATCH      ← по одному игроку
state: AWAITING_FINAL_CONFIRMATION
state: COMPLETED
state: 5MIN_CANCELLATION_WINDOW   ← после транзакции, до отправки уведомлений
```

### Решения

- ✅ Один скрин = один OCR-job в очереди (асинхронно через worker)
- ✅ Все скрины загружаются за один присест в рамках сцены
- ✅ Балансы обновляются одномоментно при подтверждении
- ✅ Уведомления — отложенно через 5 минут (окно отмены)
- ✅ Ручной ввод доступен в любой момент при ошибке OCR
- ✅ Сопоставление игроков — интерактивное, обязательное

## Сценарий 5: Создание оффлайн-сессии

**Триггер:** host жмёт `/newgame` → выбирает "🪑 Оффлайн".

### Шаги

```
Шаг 0. Та же проверка: нет ли уже активной сессии (см. сценарий 4)

Шаг 1. Параметры стола
   "Параметры стола:
   • Тип игры: [NLH] [PLO] [PLO5] [Другое]
   • Блайнды: <ввод текста '1/2'>"
   • Бай-ин мин/макс: <ввод опционально>
   • Название: <опц., дефолт 'Оффлайн от <дата>'>"
   [✓ Готово]

Шаг 2. (опц.) Co-host
   "Назначить второго админа на эту сессию?"
   [Назначить]  [Пропустить]
   ↓ (если "Назначить")
   Список членов клуба → выбор

Шаг 3. Сессия создаётся в БД
   - INSERT sessions (status='in_progress', type='offline')
   - INSERT tables (с параметрами)
   ↓
   Бот шлёт сообщение и закрепляет (pinChatMessage):
   "🎲 Сессия #42 (NLH 1/2)
   Открыта 22:14, идёт 0ч
   За столом: 0 игроков
   Общий банк: 0
   ─────────────────
   [➕ Игрок зашёл]  [💰 Докупка]
   [💵 Кэш-аут]     [📊 Стол]
   [↶ Отменить последнее]  [⏹ Завершить]"

   Если есть co-host — он получает такое же закреплённое сообщение в свой чат
```

### Состояния FSM (создание)

```
state: AWAITING_TABLE_PARAMS
state: AWAITING_COHOST_DECISION
state: AWAITING_COHOST_SELECTION
```

После создания сессии FSM **выходит из сцены**, но статус сессии в БД `in_progress`. Дальше управление — через мини-сцены сценария 6.

## Сценарий 6: Активная оффлайн-сессия (мини-сцены)

Это **не одна большая сцена**, а набор коротких мини-сцен, которые запускаются нажатием кнопок на закреплённом сообщении.

### Мини-сцена "Игрок зашёл"

```
"Кто заходит?"
[Список членов клуба]  [➕ Гость]  [⏪ Отмена]

  • Член клуба → выбор
  • "Гость" → "Имя гостя?" → ввод

"На какую сумму бай-ин?"
─ ввод числа

→ INSERT session_participants (если ещё не было)
→ INSERT buy_in_events
→ Закрепленное сообщение обновляется (editMessageText)
→ "✅ Вася зашёл на 500"
```

### Мини-сцена "Докупка"

```
"Кто докупается?"
[Список тех, кто за столом, кэш-аут не делал]

"На сколько?"
─ ввод числа

→ INSERT buy_in_events
→ Обновление pin
→ "✅ Вася докупил +300"
```

### Мини-сцена "Кэш-аут"

```
"Кто выходит?"
[Список тех, кто за столом]

"С каким стеком выходит?"
─ ввод числа

→ INSERT cash_out_events
→ Обновление pin
→ "✅ Вася вышел со стеком 1200, P&L: +400"
```

### Мини-сцена "Отменить последнее"

```
"Последнее событие сессии #42:
💰 Докупка
Игрок: Вася
Сумма: +300
Время: 23:14 (5 минут назад)

Отменить?"
[✅ Да, отменить]  [❌ Нет]

→ UPDATE buy_in_events SET deleted_at=NOW(), deleted_by=host (soft-delete)
→ Обновление pin
→ "✅ Отменено: Вася докупил +300"
```

### Мини-сцена "📊 Стол" (просмотр текущего состояния)

```
"📊 Текущее состояние стола

За столом сейчас: 4 игрока
• Вася (зашёл 500, докупил 300, в стеке: ?)
• Петя (зашёл 600, в стеке: ?)
...

Уже вышли:
• Маша (зашла 500, вышла 800, P&L: +300)

Общий банк: 2800
Бай-инов всего: 3600
Кэш-аутов: 800"
[⬅ Назад]
```

### Edge cases

1. **Игрок ушёл, потом вернулся** — при попытке "Игрок зашёл" → выбор Васи → бот спрашивает "Вася уже выходил со стеком 800. Это новое возвращение?":
   - "Да, новый бай-ин" → новый buy_in_event для существующего session_participant
   - "Нет, отменить кэш-аут" → soft-delete предыдущего cash_out_event

2. **Случайное нажатие** — после INSERT можно отменить через "Отменить последнее" (любого типа события).

3. **Параллельные события (host + co-host)** — оба могут жать кнопки одновременно. Состояние FSM в Redis по chat_id, БД защищена транзакциями.

## Сценарий 7: Закрытие сессии

**Триггер:** host или co-host жмёт "⏹ Завершить" на закреплённом сообщении.

### Шаги

```
Шаг 1. Проверка: у всех игроков, кто за столом, есть финальный стек?
   "За столом" = есть buy_in_events без cash_out_events

   ├─ есть незакрытые → "Не могу закрыть. У Васи и Пети нет финального стека."
   │                    [Кэш-аут Васи]  [Кэш-аут Пети]  [Отмена]
   │                    → запускает мини-сцену кэш-аута, возвращает к Шагу 1
   └─ всё ок → шаг 2

Шаг 2. Итоговая таблица + проверка баланса
   Считается P&L каждого участника:
   total_buy_in = SUM(buy_in_events.amount WHERE participant_id=?, deleted_at IS NULL)
   total_cash_out = SUM(cash_out_events.stack_amount WHERE participant_id=?, deleted_at IS NULL)
   profit_loss = total_cash_out - total_buy_in

   "📊 Итог сессии #42

   • Вася   зашёл 800, вышел 1200,  P&L: +400
   • Петя   зашёл 600, вышел 200,   P&L: -400
   • Коля   зашёл 500, вышел 600,   P&L: +100
   • Маша*  зашёл 400, вышел 300,   P&L: -100  (* гость)
   ────────────────────────────────────
   Сумма P&L: 0.00 ✓
   Бай-инов: 2300, кэш-аутов: 2300

   Подтвердить?"
   [✅ Подтвердить]  [✏️ Исправить]  [❌ Отмена]

Шаг 3. (если подтверждено) Транзакция в БД:
   - UPDATE sessions SET status='ended', ended_at=NOW()
   - INSERT session_results (агрегаты по каждому участнику)
   - UPDATE club_members.current_balance += profit_loss (для членов клуба)
   - INSERT ledger_entries (reason='session_result')

Шаг 4. Постановка отложенных уведомлений (5 мин) — как в онлайн-сессии

Шаг 5. Сообщение host'у:
   "✅ Сессия #42 завершена. Балансы обновлены.
   Игроки получат уведомления через 5 минут."
   [↩️ Отменить сессию (5 мин)]  [📊 Главное меню]
```

### Edge cases

1. **Сумма P&L ≠ 0** — это значит host где-то ошибся. Бот блокирует подтверждение если |sum| > 10:
   ```
   ⚠️ Баланс не сходится: −150.00
   Бай-инов: 2300, кэш-аутов: 2150
   Куда-то ушло 150.

   Проверь данные.
   ```

2. **Конкурентное закрытие (host + co-host)** — БД-транзакция атомарна, второму говорим "Уже закрыто, тебя опередили".

## Сценарий 8: Отмена сессии в 5-минутном окне

**Триггер:** host жмёт "↩️ Отменить сессию" в течение 5 мин после закрытия.

```
↓ (подтверждено)
Транзакция:
1. UPDATE notifications SET status='cancelled' WHERE связано с session_id
   (job в BullMQ тоже отменить)

3. UPDATE club_members SET current_balance -= delta
4. INSERT ledger_entries с reason='correction', delta = -original_delta,

```

**Важно:** ledger остаётся append-only. Мы не редактируем старые записи, мы добавляем компенсирующие.

## Сценарий 19: Уведомления (worker → user)

Это **не FSM**, а поток сообщений. Все уведомления хранятся в таблице `notifications` со статусами `pending/sent/failed/cancelled`. Worker раз в минуту:

```python
SELECT FROM notifications WHERE status='pending' AND scheduled_for <= NOW() LIMIT 100;

для каждого:
   try:
       Telegram.sendMessage(user.telegram_chat_id, render_template(event_type, payload))
       UPDATE status='sent', sent_at=NOW()
   except:
       UPDATE retry_count += 1, error_message=...
       if retry_count >= 5: status='failed'
```

### Типы уведомлений

| event_type | Кому | Когда |
|---|---|---|
| `club_created` | новому host'у | CLI super-admin'а создал клуб |
| `application_received` | host'у | поступила заявка |
| `application_approved` | игроку | host одобрил |
| `application_rejected` | игроку | host отклонил |
| `session_ended` | каждому участнику | через 5 мин после закрытия |
| `dispute_opened` | всем участникам | новый диспьют |
| `dispute_resolved_affected` | затронутым | резолюция с правкой |
| `dispute_resolved_unaffected` | остальным участникам | резолюция, их P&L не изменился |
| `dispute_rejected` | автору | host отклонил диспьют |
| `balance_adjusted` | игроку | host скорректировал |
| `excluded_from_club` | исключённому | host исключил |
| `settle_up_closed` | каждому | settle-up закрыт |
| `session_inactive_warning` | host'у | сессия 6ч без активности |
| `session_auto_closed` | host'у + участникам | сессия закрыта по таймауту 12ч |
| `application_pending_reminder` | host'у | заявка > 48ч без ответа |
| `dispute_pending_reminder` | host'у | диспьют > 48ч без ответа |

### Cron-задачи воркера

| Задача | Расписание | Что делает |
|---|---|---|
| `check_inactive_sessions` | каждые 30 мин | находит активные сессии без активности 6ч+ → создаёт `session_inactive_warning` |
| `auto_close_sessions` | каждые 30 мин | закрывает активные сессии без активности 12ч+ |
| `remind_pending_applications` | каждый час | напоминает host'ам о заявках > 48ч |
| `remind_pending_disputes` | каждый час | напоминает host'ам о диспьютах > 48ч |
| `auto_reject_old_applications` | раз в сутки | заявки > 7 дней → status='rejected' |
| `dispatch_notifications` | каждую минуту | разбирает очередь notifications |
