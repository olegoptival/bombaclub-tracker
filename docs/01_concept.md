# 01. Концепция

## Проблема

Компания друзей регулярно играет в покер — иногда онлайн через ClubGG, иногда оффлайн на живых столах. Игра идёт **на жетоны**, не на деньги. По итогам периода (день/неделя/месяц) нужно понимать, кто кому сколько должен.

Сейчас это решается записями в общем чате, что неудобно:
- легко потерять данные между сессиями
- нельзя посмотреть свою статистику
- расчёт "кто кому переводит" ручной, утомительный
- спорные моменты не имеют истории

## Решение

Telegram-бот, который:

1. **Записывает результаты игр** — двумя способами:
   - **Онлайн**: host загружает скрин(ы) из ClubGG → OCR через Claude Vision → бот распознаёт игроков и P&L → host подтверждает → данные в БД
   - **Оффлайн**: host открывает живую сессию, по ходу вводит события (бай-ин, докупка, кэш-аут), в конце закрывает

2. **Ведёт накопительный баланс** каждого игрока в клубе через append-only ledger (полная история изменений).

3. **Считает settle-up** — кто кому сколько должен с минимизацией числа переводов (точный алгоритм).

4. **Поддерживает споры** — игрок может оспорить результат, host разбирает.

5. **Уведомляет участников** через Telegram о завершении сессий и settle-up.

## Целевая аудитория

**MVP**: **наша компания друзей.** Закрытая система с приглашениями только от host'а.

**Потенциально**: SaaS для других покерных клубов. Но это **только потом** — архитектурно заложено (мультитенантность через `club_id`), но фокус сейчас на работающем продукте для своих.

## Что НЕ делаем

- ❌ Игру в покер (не покерный клиент). Только учёт.
- ❌ Реальные деньги, эквайринг, KYC.
- ❌ Открытую регистрацию. Только инвайты.
- ❌ Веб-фронт на этапе MVP. Только Telegram-бот.
- ❌ Интеграцию с ClubGG API (его нет, скрейпинг — серая зона). Только OCR со скринов.

## Ключевые продуктовые решения

| Что | Решение | Почему |
|---|---|---|
| Создание клубов | CLI super-admin'а | Спам-защита, не размениваться на UX в боте |
| Регистрация игроков | Только по инвайту от host'а + подтверждение | Закрытая система, доверие |
| Жетоны | Просто число, не "деньги" | Нет регулирующих требований |
| Округление в settle-up | Не делаем | Сложность не оправдана пользой |
| Период settle-up | Только ручной запуск host'ом | Нет cron-сложностей |
| Балансы при settle-up | Обнуляются | Семантика "все рассчитались" |
| Гости в сессии | Поддерживаются (без баланса в клубе) | Реальные кейсы — приходят знакомые |
| Округление переводов | Игроки сами при переводе | Не наша задача |

## Принципы разработки

1. **Скучный стек.** TypeScript + Postgres + Redis. Никакой экзотики.
2. **Одно решение — один файл.** Концентрация логики, не размазывать.
3. **Append-only ledger.** История всех изменений баланса не редактируется никогда.
4. **Бот тонкий, worker толстый.** Бот моментально отвечает, тяжёлые задачи — в очередь.
5. **Веб опционален.** Каркас под него заложен (`src/api/`), но для MVP не нужен.
