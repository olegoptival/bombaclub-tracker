# 04. Журнал решений

Здесь зафиксированы **все ключевые развилки** и почему выбран тот или иной вариант. Полезно вернуться, когда забудешь, почему сделано именно так, или когда захочешь поменять — увидишь, какие соображения были.

## Архитектура

### А1. Бот, не веб
**Решение:** только Telegram-бот на этапе MVP, без веб-фронта.
**Альтернативы:** PWA, нативное мобильное приложение, веб + бот.
**Почему так:**
- Telegram уже идентифицирует пользователя (chat_id) — нет регистрации, паролей, восстановления.
- Уведомления нативные.
- Деплой на одном VPS, никакого мобильного store.
- Веб можно добавить потом — слой `src/api/` заложен.

### А2. TypeScript / Node.js
**Решение:** TypeScript на всём бэкенде (бот, worker, будущий API).
**Альтернативы:** Python, Go.
**Почему так:**
- grammy — лучшая bot-библиотека для Node.
- Один язык на всё, переиспользование типов и логики.
- TypeScript типы спасают в FSM, где сложные state machines.

### А3. Polling по дефолту, webhook при появлении домена
**Решение:** на старте — polling (без HTTPS, без домена).
**Почему так:**
- Можно стартовать с любого VPS моментально.
- Переключение на webhook — одна переменная окружения.
- Caddy выносится в отдельный compose-overlay, не мешает базовой установке.

### А4. Бот тонкий, worker толстый
**Решение:** OCR, settle-up, уведомления, cron — всё в worker. Бот только UI.
**Почему так:**
- Telegram даёт ~30 сек на ответ webhook'у. OCR занимает 3-7 сек.
- Бот должен отвечать моментально ("обрабатываю...") и не блокироваться.
- Воркер можно скейлить независимо.

### А5. PostgreSQL, не NoSQL
**Решение:** Postgres + Prisma.
**Альтернативы:** MongoDB, SQLite.
**Почему так:**
- Сильно реляционные данные (clubs, members, sessions, ledger).
- Нужны транзакции для обновления балансов.
- Нужны UNIQUE constraints, partial indexes (например, "одна активная сессия на host'а в клубе").

### А6. Append-only ledger
**Решение:** все изменения баланса = новая запись в `ledger_entries`. Никогда не редактируем и не удаляем (триггер БД блокирует UPDATE/DELETE).
**Почему так:**
- Полная аудитируемость "почему у игрока такой баланс".
- Ошибки исправляются компенсирующими записями, а не правкой истории.
- Доверие в клубе: никто не может незаметно подкрутить.

### А7. MinIO для скринов, не S3
**Решение:** локальный MinIO в docker-compose.
**Альтернативы:** AWS S3, Backblaze B2, Cloudflare R2.
**Почему так:**
- Пользователь явно попросил локальное хранение.
- Нет внешних счетов и зависимостей.
- На уровне кода — тот же S3 SDK, миграция в облако (если когда-то) тривиальна.

## Концепция и UX

### К1. Создание клуба — CLI, не в боте
**Решение:** клубы создаёт super-admin через `npm run create-club -- ...`.
**Альтернативы:** открытая регистрация в боте; /create_club доступная только super-admin.
**Почему так:**
- Открытая регистрация = спам, мусор, нагрузка.
- Реализация в боте требует много UX (выбор настроек, кнопки) — для разовой задачи неоправданно.
- Когда появится веб — там будет нормальная админка.
- Для MVP "клуб друзей" — клуб создаётся 1 раз и забывается.

### К2. Регистрация через инвайт + подтверждение host'ом
**Решение:** только по ссылке от host'а, после прохождения сцены — заявка ждёт подтверждения.
**Альтернативы:** автоматическое вступление по ссылке; открытая регистрация.
**Почему так:**
- Закрытая система = доверие.
- Двойная проверка: правильный ник, ClubGG привязан, host видит кто заявился.
- Защита от случайных проходов по ссылке.

### К3. Инвайт = 24 часа, одноразовый
**Решение:** короткий TTL, один инвайт = один человек.
**Альтернативы:** 7 дней, многоразовый.
**Почему так:**
- Безопасность важнее удобства.
- Если инвайт многоразовый и попадает в общий чат — потерял контроль.
- 24 часа — host обычно отдаёт ссылку лично, в день встречи.

### К4. Жетоны = просто число
**Решение:** decimal без курса к деньгам, без эмиссии, без подарков в схеме.
**Почему так:**
- Подарки и обмен — отдельный модуль на потом.
- Никаких регулирующих требований.
- Простая семантика: "у тебя N жетонов в клубе".

### К5. Округление в settle-up — не делаем
**Решение:** всегда точные суммы до копеек.
**Альтернативы:** настраиваемое округление (10/50/100), накопление дельт.
**Почему так:**
- Слишком сложно для вознаграждения.
- Реальные люди округляют сами при переводе ("давай я тебе 300 переведу" вместо 293.33).
- Дельты от округления требуют отдельной логики и таблиц.

### К6. Балансы обнуляются при закрытии settle-up
**Решение:** после закрытия периода current_balance = 0 у всех (через ledger).
**Альтернативы:** настраиваемое поведение; накопительно всегда.
**Почему так:**
- Семантика "settle-up закрыт = все рассчитались" — естественная.
- Если игрок недоплатил — это уже проблема между ним и кредитором, вне системы.
- Чистый старт следующего периода.

### К7. Settle-up только ручной, без cron
**Решение:** host жмёт кнопку когда хочет.
**Альтернативы:** cron каждый день/неделю, гибрид.
**Почему так:**
- Нет проблемы "пропустили срабатывание" / "что если идёт сессия в момент cron".
- Host лучше понимает "пора" — например, конец вечеринки.
- Хочешь напоминание? Можно добавить уведомление "прошла неделя без settle-up" (не сделано в MVP).

### К8. Игроки не подтверждают переводы
**Решение:** после settle-up игрок просто видит "тебе нужно перевести X / тебе должны Y".
**Альтернативы:** кнопки "я перевёл" / "получил".
**Почему так:**
- Реальные переводы — вне нашей системы (наличные, банк).
- Если игрок не перевёл — это разбирают сами участники.
- Не усложняем UX для случая, который и так очевиден.

## Сессии

### С1. Онлайн-сессия = "один присест"
**Решение:** все скрины загружаются в одной сцене, нельзя добавить позже.
**Альтернативы:** сессия живёт долго, скрины приходят постепенно.
**Почему так:**
- Меньше состояний.
- Меньше edge cases ("сессия открыта 2 дня, что с балансами?").
- Скрины ClubGG приходят пачкой — это естественная модель.

### С2. Один скрин ≠ один стол; один скрин = один раунд стола
**Решение:** **сумма P&L по всем скринам** = итог сессии. Несколько скринов одного стола ClubGG = несколько раундов, P&L складываются.
**Альтернативы:** primary screen + archives (мы это рассмотрели и выкинули).
**Почему так:**
- Это **реальная семантика ClubGG**: стол живёт 12 часов, может быть несколько игровых сессий внутри.
- Тест на реальных скринах: суммы P&L двух скринов сходятся в ноль независимо, агрегат тоже = 0. Проверено.

### С3. Гости поддерживаются в обоих типах сессий
**Решение:** гость = `session_participants.guest_name` без `club_member_id`. Не имеет накопительного баланса в клубе.
**Альтернативы:** только члены клуба; гости как полноценные user'ы временно.
**Почему так:**
- Реальный кейс: пришёл знакомый поиграть один раз.
- Создавать ему membership — лишний шум и баланс висит впустую.
- В settle-up гости не участвуют (нет в `club_members`).

### С4. Оффлайн-сессия = живая, события поштучно
**Решение:** не "список игроков с бай-инами", а поток `buy_in_events / cash_out_events`.
**Альтернативы:** агрегированный ввод в конце.
**Почему так:**
- Сохраняется история: 3 бай-ина и 1 кэш-аут vs один итог.
- Можно отменить последнее событие.
- Можно посмотреть текущее состояние стола.
- Игрок ушёл и вернулся — естественно: новый buy-in event.

### С5. Auto-таймаут оффлайн-сессии = 12 часов
**Решение:** напоминание в 6ч, авто-закрытие в 12ч.
**Альтернативы:** 24ч, 48ч, без auto-close.
**Почему так:**
- Защита от "вечных сессий", когда host забыл закрыть.
- 12ч — больше любой реальной игры.
- При авто-закрытии host получает уведомление и может оспорить.

### С6. Закрытие требует финальные стеки у всех
**Решение:** нельзя закрыть сессию пока у кого-то нет cash-out (если он за столом).
**Почему так:**
- P&L нельзя посчитать без финального стека.
- Если игрок ушёл и забыли — баланс не сойдётся.

### С7. 5-минутное окно отмены после закрытия
**Решение:** сразу после закрытия сессии — кнопка "Отменить" доступна 5 минут. Через 5 минут уведомления улетают игрокам, кнопка пропадает.
**Альтернативы:** мгновенная отправка уведомлений; больший/меньший buffer.
**Почему так:**
- Защита от опечатки на финальном экране.
- Уведомления через 5 мин — нормальная практика (Slack так же делает).
- После 5 мин правки идут только через диспьюты.

### С8. Отмена сессии = compensating ledger entries
**Решение:** при отмене НЕ удаляем `ledger_entries` оригинальной сессии, а добавляем компенсирующие с `reason='correction'`.
**Почему так:**
- Ledger остаётся append-only.
- В истории видно: сессия была, потом отменена.
- Можно потом расследовать "что произошло".

## OCR

### O1. Claude Vision как провайдер OCR
**Решение:** `claude-sonnet-4-5` через Anthropic API.
**Альтернативы:** GPT-4o Vision, Google Cloud Vision, Tesseract.
**Почему так:**
- Возвращает сразу структурированный JSON, не raw text.
- Хорошо работает с разными языками (русский, иврит, китайский в никах).
- Понимает контекст ("это таблица результатов покера").
- Стоимость $0.012 / скрин — копейки для нашего объёма.
- Tesseract / Google Vision требуют постпроцессинг координат — лишний код.

### O2. Гибридные пороги расхождения баланса
**Решение:**
- |sum| ≤ 0.01 → OK тихо
- 0.01 < |sum| ≤ 1 → INFO в логи (округления ClubGG)
- 1 < |sum| ≤ 10 → WARN: разрешаем подтвердить с предупреждением
- |sum| > 10 → BLOCK: подтверждение запрещено

**Почему так:**
- Полностью блокировать = слишком жёстко (1-2 копейки бывают от округлений в клубе).
- Полностью разрешать = опасно (большое расхождение = OCR пропустил игрока).
- Гибрид даёт баланс.

### O3. temperature=0
**Решение:** жёсткий детерминизм в OCR-вызовах.
**Почему так:**
- Парсинг должен быть повторяем.
- Не нужна "креативность".

### O4. Хранение скринов 90 дней в MinIO
**Решение:** загруженные скрины храним 90 дней, потом cron-чистка.
**Альтернативы:** хранить вечно, не хранить вообще, настраиваемое.
**Почему так:**
- 90 дней — больше окна диспьютов (7 дней) и settle-up периода (max месяц).
- Места не сильно жалко (MinIO локально).
- Можно сделать настройкой клуба позже.

### O5. Дедупликация скринов в рамках сессии
**Решение:** UNIQUE `(session_id, image_hash)` — нельзя загрузить тот же файл дважды в одну сессию.
**Почему так:**
- Защита от случайной двойной отправки.
- Идемпотентность.
- Один и тот же файл в РАЗНЫХ сессиях разрешён (валидно).

### O6. Сопоставление игроков — интерактивное, обязательное
**Решение:** все нераспознанные игроки — host выбирает из списка членов или добавляет гостем.
**Альтернативы:** auto, вручную потом.
**Почему так:**
- Атомарность: сессия записывается с правильно сопоставленными игроками сразу.
- "Auto = всё гости" приведёт к проблемам с балансами.
- При сопоставлении создаётся `member_aliases` — следующий раз авто-сопоставление сработает.

## Безопасность и права

### П1. Три роли: player / host / superuser
**Решение:** на уровне клуба — player и host. На уровне системы — superuser (глобально).
**Почему так:**
- Простая модель.
- Roles per club: один человек может быть host'ом одного и player'ом другого.
- Co-host пока на уровне сессии, не клуба.

### П2. Host исключает игрока с любым балансом
**Решение:** разрешено, но с подробным предупреждением о последствиях.
**Альтернативы:** блокировать пока баланс не = 0.
**Почему так:**
- Иногда реально нужно (игрок пропал, не выходит на связь).
- Бот предупреждает что баланс клуба перестанет сходиться к 0.
- Host берёт ответственность.

### П3. Корректировка баланса host'ом — без лимита
**Решение:** host может изменять баланс на любую сумму.
**Альтернативы:** лимит 1000 за раз, требовать super-admin для больших изменений.
**Почему так:**
- Закрытое сообщество, host = человек которому доверяют.
- Все правки в ledger с обязательной причиной (мин 10 символов) и уведомлением игроку.
- Игрок может оспорить через диспьют.

### П4. Игроки видят чужие балансы только в settle-up и в сессиях, где играли вместе
**Решение:** в `/balance` — только свой. В сессии и settle-up — балансы всех участников.
**Почему так:**
- Privacy без излишеств.
- Игрок имеет право знать с кем играл и кто сколько проиграл/выиграл в общей сессии.

### П5. Диспьюты видны всем участникам сессии
**Решение:** при открытии диспьюта все участники сессии получают уведомление.
**Альтернативы:** только host и автор диспьюта.
**Почему так:**
- Прозрачность: все знают что возможны правки.
- Если правка коснётся не только автора — другие не удивятся.

## Что НЕ делаем (антирешения)

| Что | Почему НЕ |
|---|---|
| ClubGG API интеграция | Его нет, скрейпинг — серая зона. OCR покрывает 95% |
| Hand history парсинг | Лишняя сложность для нашей задачи. P&L достаточно |
| Открытая регистрация | Спам и нагрузка |
| Cron settle-up | Лишний класс багов "пропустили срабатывание" |
| Округление переводов | Игроки сами справляются |
| Подарки/обмен жетонов | Отдельный модуль на потом |
| Telegram Mini Apps | Пока хватит обычного бота |
| Метрики (Prometheus) | Pino-логов и docker logs хватает на старте |
| Kubernetes | Оверкилл, docker-compose решает |
| Multi-instance бота | Telegram webhook'и плохо дружат с горизонтальным масштабированием |
| Staging-окружение | Side-project с 10 пользователями — избыточно |
